# 21 DB-First Identifier Cutover

## Objective

รีเซ็ตแนวทางงาน identifier migration ให้เป็น `DB-first` อย่างชัดเจน:

- `id` ต้องเป็น internal primary key / foreign key เท่านั้น
- `id` ของ master / transaction หลักต้องไปทาง `bigint generated by identity`
- `code` หรือ `doc_no` ต้องเป็น business identifier ที่ใช้ใน UI/API/import/export/filter/report
- ห้ามใช้ fallback จาก business/display field กลับไป internal `id` เพื่อประคอง runtime

เอกสารนี้เป็น log แยกสำหรับงาน cutover ฐานข้อมูล และเป็นตัวคุม wave ของ migration ชุดนี้

## Target Rules

### Canonical Identifier Shape

- master tables:
  - `id bigint generated by identity`
  - `code text unique` เป็น business key
- document tables:
  - `id bigint generated by identity`
  - `doc_no text unique` เป็นเลขเอกสารธุรกิจ
- transaction refs:
  - relational columns เช่น `branch_id`, `supplier_id`, `customer_id`, `warehouse_id`, `sales_id` ต้องอ้าง `bigint id`
- outward contract:
  - UI/API/filter/export ต้องอ้าง `code` หรือ `doc_no`
  - internal `id` ห้ามหลุดเป็น business/display fallback

### Execution Policy

- ทำแบบ `schema-first`
- ปรับ DB ให้ถึง target shape ก่อน
- จากนั้นค่อยไล่แก้ Prisma/runtime/code ให้ conform กับ schema จริง
- ถ้ามีจุดพังใน dev ให้แก้ตรงจุดนั้น ไม่เพิ่ม fallback debt
