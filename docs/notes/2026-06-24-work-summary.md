# บันทึกความคืบหน้าการทำงาน (24 มิถุนายน 2026)

การทำงานในรอบนี้มุ่งเน้นการซ่อมบำรุงและทดสอบ UAT จริงของระบบ **LINE Notification** สำหรับแจ้งเตือนใบชั่งน้ำหนัก (Weight Ticket WTI/WTO) ให้สามารถกระจายข้อมูลเข้าสู่กลุ่มไลน์ได้สมบูรณ์และพรีเมียม

---

## 📂 สรุปคำอธิบายเชิงธุรกิจและการออกแบบ (Flow & Rationale Summary)

### 1. เอนทิตีทางธุรกิจที่เกี่ยวข้อง (Business Entities)
- **`system_settings` (การตั้งค่าระบบ):** เก็บข้อมูลความลับและข้อมูลกำหนดค่าส่วนกลางของระบบ LINE Messaging API เช่น `LINE_CHANNEL_ACCESS_TOKEN`, `LINE_CHANNEL_SECRET`, และชื่อถังเก็บข้อมูล PDF `WEIGHT_TICKET_PDF_BUCKET`
- **`line_groups` (เส้นทางการกระจายแจ้งเตือน):** ทำหน้าที่เป็นตารางเก็บความสัมพันธ์การกระจายข้อความรายกลุ่มไลน์ที่ลงทะเบียนไว้ผ่าน Webhook โดยผู้ใช้สามารถกำหนดเงื่อนไขได้ว่ากลุ่มนี้จะให้ทำงานเฉพาะสาขาใด (`branch_code`) และระบุสวิตช์รับข้อมูลเอกสารรับเข้า (`notify_wti`) หรือเอกสารส่งออก (`notify_wto`)
- **`weight_ticket_notification_logs` (ประวัติการส่งข้อมูล):** ทำหน้าที่บันทึกทรานแซกชันการแจ้งเตือนเพื่อใช้เป็นข้อมูลตรวจสอบย้อนกลับ (Audit Log) รวมถึงป้องกันการแจ้งเตือนซ้ำซ้อน

### 2. ตรรกะเบื้องหลังการทำงาน (Design Rationale & Logic)
- **ทำไมต้องส่ง Flex Message โครงสร้างนี้:** โครงสร้าง Flex Message ถูกปรับปรุงให้มีรายละเอียดทางการเงินและน้ำหนักครบถ้วนที่สุดตามความต้องการจริงของลูกค้า โดยมีหัวเรื่องชัดเจนตามประเภทธุรกรรม (เช่น ใบรับของ WTI / ใบส่งของ WTO) แสดงน้ำหนักสุทธิสีเขียว/ฟ้า เพื่อช่วยให้อ่านข้อมูลสรุปได้อย่างชัดเจนรวดเร็วบนแชทไลน์ และมีปุ่มนำทางไปยัง PDF และหน้าเว็บระบบหลักโดยตรง
- **ระบบป้องกันการส่งข้อความซ้ำซ้อน (Duplicate Prevention):** ในหน้าจองานปกติ พนักงานอาจมีการกดอัปเดตและบันทึกใบชั่งซ้ำหลายครั้ง ซึ่งอาจทำให้ข้อความแจ้งเตือนไหลเข้ากลุ่มไลน์ของลูกค้าซ้ำซาก ระบบจึงนำคิวรี่จากตาราง `weight_ticket_notification_logs` มาบล็อกการส่งซ้ำอัตโนมัติหากกลุ่มนั้นเคยได้ข้อความเดิมแล้ว ยกเว้นกรณีพนักงานตั้งใจกด "แชร์ส่งเข้ากลุ่ม" บนหน้าจอโดยตรง (Manual Share) ซึ่งจะใช้พารามิเตอร์ `force: true` บังคับส่งใหม่ทันที

## 🛠️ งานที่ดำเนินการสำเร็จแล้วในรอบนี้

### 1. แก้ไขปัญหารูปแบบ Flex Message ผิดพลาด (LINE API Payload Fix)
* **สาเหตุของปัญหา:** ฟังก์ชัน `buildFlexMessage` ส่งคืนออบเจ็กต์ข้อมูลโดยลืมระบุฟิลด์ `"type": "flex"` ที่ระดับบนสุดของ payload ส่งผลให้ LINE API ส่งรหัสข้อผิดพลาดกลับมาเป็น 400 Bad Request (`messages[1].type: May not be empty`) ในขณะส่งจริง
* **การแก้ไข:** ทำการเพิ่มคีย์ `type: 'flex' as const` ลงในออบเจ็กต์ผลลัพธ์ในไฟล์ [weight-ticket-line-notification.ts](file:///c:/new-ns-scrap-erp/apps/next/src/lib/server/weight-ticket-line-notification.ts) ทำให้โครงสร้างข้อความถูกต้องและส่งผ่าน API สำเร็จ

### 2. พัฒนาและรันสคริปต์จำลอง UAT ยิงแจ้งเตือนจริงสำเร็จ (Local UAT Simulation)
* **การจำลองสิทธิ์ PDF:** เนื่องจากเครื่องพัฒนา Local ไม่มีสิทธิ์ RLS คีย์ `SUPABASE_SERVICE_ROLE_KEY` สำหรับการเขียน/อัปโหลดไฟล์จริงขึ้น Supabase Storage ผมจึงเพิ่มระบบทดสอบจำลองผ่านตัวแปรแวดล้อม `MOCK_PDF_UPLOAD=true` เพื่อให้สามารถทำงานข้ามส่วน RLS และดำเนินส่วนเชื่อมต่อ LINE API ต่อได้
* **การลงทะเบียนและสลับการตั้งค่ากลุ่มทดสอบ:** 
  - ทำการรันสคริปต์ [setup_test_group.ts](file:///c:/new-ns-scrap-erp/apps/next/scratch/setup_test_group.ts) ลงทะเบียนกลุ่มเป้าหมาย **`C811b8c9acfcf27f649039be5055e32fb`** ลงในตาราง `line_groups`
  - ทำการเคลียร์ค่า `LINE_DEFAULT_TARGET_ID` เป็นค่าว่าง เพื่อกระตุ้นให้ระบบทำงานผ่าน Multi-target / Branch Routing
* **ยิงแจ้งเตือนจริงสำเร็จ:** รันสคริปต์ทดสอบยิงข้อมูลใบชั่งล่าสุด `WTI012606-0022` ผลปรากฏว่า **LINE API ตอบกลับ 200 OK ส่งสำเร็จเรียบร้อย** ได้ข้อความ summary และ Flex Message เข้าในกลุ่มทดสอบเรียบร้อย
