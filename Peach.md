# Peach UI Guidelines & Reference (AcexPOS Style)

เอกสารนี้รวบรวมข้อกำหนดการออกแบบและการปรับปรุงระบบอินเตอร์เฟส (UI Rehabilitation) ที่เกิดขึ้นในบรันช์ `peach` (Commit: `7223e7a` เมื่อ 2026-06-13) เพื่อใช้เป็นแนวทางมาตรฐานในการพัฒนาหน้าจอการแสดงผลอื่น ๆ ของระบบ **NS Scrap ERP** ให้มีลักษณะเดียวกับสไตล์ **AcexPOS**

---

## 🎨 หลักการดีไซน์สำคัญ (AcexPOS UI Standard)

### 1. โครงสร้างการ์ดข้อมูลและตัวชี้วัด (KPI / Metric Summary Cards)
สำหรับกล่องตัวเลขสถิติหรือสรุปสถานะด้านบนของหน้าเว็บ (KPI Overview):
- **🚫 ห้ามมีการ์ดด้านหลัง (No Outer Wrapper Card):** ห้ามใช้กล่องครอบนอก (Outer Wrapper) เช่น `bg-slate-50 border-slate-200` ครอบทับด้านหลังกลุ่มการ์ดสถิติเด็ดขาด เพื่อไม่ให้ดูเป็นการ์ดซ้อนทับกันหลายชั้นตามที่ผู้ใช้กำหนด ให้จัดวางการ์ดสถิติต่าง ๆ ลงบน Grid เปล่า ๆ และเรนเดอร์บนพื้นหลังหลักโดยตรง
- **การ์ดข้อมูลหลัก (Metric Cards):**
  - สไตล์: `bg-white shadow-sm border border-slate-200 rounded-xl`
  - ตกแต่งไอคอนด้านซ้าย: ใช้ไอคอนพร้อมพื้นหลังวงกลมสีพาสเทลอ่อน (Circular Pastel Background) เพื่อให้ดูพรีเมียมและสวยงาม
  - ตัวเลขเด่นชัด และมีชื่อกำกับชัดเจนอยู่ทางด้านขวา
  - **🚫 ข้อห้ามสำคัญ:** ห้ามใช้รูปแบบ Left-border indicator / Left-border Accent (การ์ดที่มีแถบสีขอบด้านซ้ายหนา) โดยเด็ดขาด เพราะทำให้ขาดความเป็นระเบียบและไม่สวยงาม
  - **⚖️ ตรรกะสีตัวเลขทางการเงิน:** ในกรณีการ์ดที่แสดงผลต่างหรือ Margin หากค่าตัวเลขเป็นศูนย์ (`0.00` หรือไม่มีส่วนต่าง) ต้องแสดงผลในโทนสีเทาที่เป็นกลาง (`Neutral Slate` เช่น ไอคอนใช้ `bg-slate-100 text-slate-600` และตัวเลขใช้ `text-slate-900`) ห้ามใช้สีเขียวหรือแดงพร่ำเพรื่อเพื่อป้องกันการตีความคลาดเคลื่อนทางการเงิน

### 2. การจัดวางแถวตัวกรองและปุ่มปฎิบัติการ (Toolbar / Filters layout)
- **การปัดขวาของปุ่มหลัก (Desktop Actions):** ปุ่มคำสั่งหลัก (เช่น Refresh, Export Excel, หรือ เพิ่มรายการใหม่) บนหน้าจอ Desktop ให้ผลักไปอยู่มุมบนขวาของแถวตัวกรอง (Filter Row) โดยใช้คลาส Tailwind `ml-auto`
- **การตอบรับบน Mobile:**
  - ใช้ **Filters Drawer** หรือ **Bottom Sheet** สำหรับหน้าจอขนาดเล็กเพื่อซ่อนตัวกรองที่เยอะเกินไป
  - แยก Toolbar ระหว่าง Desktop และ Mobile ให้มีสไตล์เฉพาะตัว

### 3. ตารางข้อมูลหลัก (Data Table & Grid)
- **การใช้ตารางตามรูปแบบ /purchase/bills (Table Lined View):** ตารางแสดงข้อมูลหลักบน Desktop ให้ใช้รูปแบบที่มีเส้นคั่นแถวบางเบา (`divide-y divide-slate-100`) และใช้ขอบขอบกล่องสีเทาอ่อนนุ่มนวล (`border border-slate-200 shadow-sm rounded-md`) หลีกเลี่ยงเส้นขอบดำหนา และมีพฤติกรรม hover พื้นหลังแบบนุ่มนวล (`hover:bg-slate-50`) เช่นเดียวกับหน้า `/purchase/bills`
- **การปรับความกว้างและการเรียงลำดับคอลัมน์ (Resizable & Sorting Columns):** ทุกตารางหลักและตารางย่อยที่แสดงข้อมูลในโหมด Desktop ต้องรองรับฟังก์ชันการยืดหดความกว้างคอลัมน์ (Resizable Columns โดยใช้ `useResizableColumns` และ `<ResizableTableHead>`) และรองรับการเรียงลำดับข้อมูล (Sorting) เพื่อปรับปรุงความยืดหยุ่นในการใช้งาน โดยต้องมีปุ่ม "คืนค่าเดิมตาราง" (Reset Widths) เสมอ
- **การยุบตารางบนมือถือ:** ตารางขนาดใหญ่จะถูกซ่อนบนหน้าจอขนาดเล็ก (`hidden md:table` หรือ `hidden lg:table`) และแทนที่ด้วยรายการการ์ดกระชับ (Mobile Dense Card List) ที่สามารถเลื่อนอ่านได้สะดวก
- **ลบ Focus Outline:** ปิดแถบขอบดำที่เกิดจากการกดเลือกของเบราว์เซอร์ดีฟอลต์ (Focus Outline) ในส่วนต่าง ๆ เช่น ปุ่ม และอินพุต โดยการเพิ่มคลาส `outline-none` หรือการกำหนดสไตล์โกลบอลใน `globals.css`


### 4. ป๊อปอัปและฟอร์มทำรายการ (Modals / Forms Layout)
- **หัวข้อบาร์แบบเข้ม (Dark Header):** ฟอร์มหลักและกล่องรายละเอียดข้อมูล (Modal/Dialog) ให้ใช้แถบหัวเรื่องเป็นสีเข้ม `bg-slate-900` ร่วมกับข้อความสีขาว `text-white`
- **ส่วนท้ายฟอร์ม (Form Footer):**
  - ปุ่มยกเลิกและปุ่มยืนยันให้จัดชิดขวาร่วมกัน
  - ปุ่มยกเลิกใช้รูปแบบข้อความเรียบง่ายไร้กรอบ (Text-only Button) และปุ่มตกลง/บันทึกใช้ดีไซน์หลัก เช่น สีเด่นมีน้ำหนักมนสวยงาม (เช่น `bg-[#0F172A] hover:bg-[#1E293B]`)

### 5. สรุปความต้องการเฉพาะ "แบบที่ผมชอบ" (User's Preferred Style Details)
- **ห้ามมีเส้นขอบสีดำ (No Black Lines/Borders):** ทั้งในตาราง (Tables), ช่องกรอกข้อมูล (Inputs), ปุ่มกด (Buttons) หรือกล่องต่าง ๆ ห้ามใช้เส้นขีดสีดำเข้มหรือสีดำเด็ดขาด ให้ใช้สีพาสเทลอ่อนนุ่มนวล เช่น `border-slate-100` หรือ `border-slate-200/60` (สำหรับตาราง) และ `border-slate-300` (สำหรับฟอร์มและตัวกรอง)
- **ห้ามมีขอบขาว/กรอบเส้นขีดรอบ Dialog Modal:** ตัว Modal (เช่น `DialogContent`) ห้ามมีเส้นขอบสีเทาหรือสีขาวสว่างล้อมรอบตัว Modal (ให้เอาคลาส `border` หรือ `border-slate-200` ออก และเหลือเฉพาะเงาหนา `shadow-2xl` หรือเงาประเภทนุ่มนวลเพื่อแบ่งมิติขอบเขตแทน) เพื่อให้หัวบาร์สีเข้มและส่วนบอดี้ดูกลมกลืนและพรีเมียม ไร้กรอบขีดขาวสว่างรอบหัว Modal สีดำ
- **การ์ด KPI / Decision Signals บนหน้าจอขนาดเล็ก:** บนหน้าจอมือถือ/แท็บเล็ต หรือ viewport ที่มีพื้นที่ว่างด้านขวาเหลือ ให้แสดงเป็น 2 คอลัมน์คู่กัน (`grid-cols-2`) เสมอ เพื่อช่วยแบ่งข้อมูลไปทางด้านขวา ไม่ให้เหลือช่องว่างสีขาวขนาดใหญ่ทางขวา และช่วยลดความยาวของการเลื่อนหน้าจอลง
- **หลีกเลี่ยงตารางที่ต้องเลื่อนขวาบน Mobile & Tablet (No Horizontal Scroll Tables):** ห้ามใช้ตารางแนวนอนที่ต้องเลื่อนไปด้านขวา (Horizontal Scroll) บนอุปกรณ์ขนาดกลางและเล็กเด็ดขาด ให้แปลงตารางเป็นโครงสร้างการ์ดแนวตั้ง (Mobile Card List View) เสมอ โดยกำหนด breakpoint ที่ **`lg` (1024px)** ในการสลับเป็นแบบการ์ด เพื่อป้องกันปัญหาตารางกว้างล้นจอในกลุ่มหน้าจอขนาดกลางและ Tablet
- **ลบกรอบสีดำตอนคลิก/โฟกัส (No Browser Outline/Focus Ring):** ให้ลบ Focus Outline สีดำของเบราว์เซอร์ดีฟอลต์ออกในทุกองค์ประกอบ (เช่น ปุ่มกดปิด Modal เครื่องหมายคูณกากบาท ปุ่มเลือกเมนู) โดยการเพิ่มคลาส `outline-none` และใช้การเปลี่ยนสีไอคอนหรือ Focus Ring สีอ่อน ๆ แทน
- **ขนาดตัวอักษรและคอนโทรลตัวกรอง (Standard Font Sizing for Filters & Cards):**
  - **บน Desktop:** ตัวกรองในแถบเครื่องมือ (Toolbar Filters) ต้องไม่ใหญ่หรือเทอะทะจนเกินไป และต้องไม่เล็กจนอ่านยาก โดยให้ใช้สไตล์ความสูงธรรมชาติหรือสัดส่วนสมมาตร (ความสูงรวมประมาณ `h-9` ถึง `h-10`) ร่วมกับขนาดตัวหนังสือ `text-sm` และใช้สีขอบ `border-slate-300` พร้อมระยะห่างที่พอดี ไม่ nested กรอบสองชั้น
  - **บน Mobile:** ขนาดตัวอักษรบนการ์ดแสดงข้อมูล (Dense Card List) และหน้าตัวกรอง (Filters Drawer) ห้ามใช้ฟอนต์ขนาดเล็กเกินไป (เช่น `text-[10px]` หรือเล็กกว่า) ให้ขยายและปรับใช้ขนาด `text-xs` หรือ `text-sm` เพื่อให้อ่านออกง่าย สบายตา และมีสัดส่วนที่พอดีกับหน้าจอมือถือ
- **การใช้ Baseline Font สำหรับ UI Controls (Baseline Font Standard):** ห้ามใส่คลาส `font-sans` หรือฟอนต์อื่น ๆ ในองค์ประกอบ UI (เช่น ปุ่มกด, ปุ่มสลับสถานะ, เมนูตัวเลือก, ช่องรับข้อมูล หรือฟอร์ม) เพื่อให้ทุกส่วนแสดงผลด้วยฟอนต์เริ่มต้นหลัก `Noto Sans Thai` ที่กำหนดไว้สำหรับ body เสมอ ป้องกันความไม่สม่ำเสมอของตัวอักษร
- **การใช้ตารางข้อมูลหลักบน Desktop และการ์ดบน Mobile (Data Table & Grid Parity):** รายการเอกสารหรือรายการข้อมูลหลักบน Desktop ให้ใช้รูปแบบตารางที่มีเส้นคั่นบางเบา (`divide-slate-100`) และขอบขอบบางเบา (`border-slate-200 shadow`) สไตล์เดียวกันกับหน้า `/purchase/bills` เพื่อให้เปรียบเทียบข้อมูลสะดวก และบนอุปกรณ์หน้าจอขนาดเล็ก (Mobile/Tablet) ให้ยุบแสดงเป็นรายการการ์ดกระชับ (Mobile Card List View) เสมอ เพื่อความ Responsive

## ⚠️ ข้อควรระวังเพิ่มเติมในการพัฒนา UI
* **ห้ามมีกรอบขอบดำรอบปุ่มหรืออินพุตเมื่อถูกเลือก:** ตรวจสอบให้มั่นใจว่าปุ่มใช้งานไม่มี Browser Focus Ring สีดำหนา
* **การออกแบบ Modal:** ต้องครอบคลุมพฤติกรรมบนอุปกรณ์เคลื่อนที่ด้วยเสมอ โดยเฉพาะเรื่องความสูงของหน้าจอและระยะสัมผัส (Touch Targets)
* **การจัดการรูปสินค้า:** ให้ใช้มาตรฐาน Storage bucket `product-images` ในการจัดเก็บรูปภาพสินค้า (อ้างอิงจาก Standard ใหม่ในระบบคลัง)

## 🚫 กฎควบคุมการทำงาน (Git & Scope Rules)
* **ห้ามเพิ่มข้อมูลสินค้าหรือข้อมูลหลักเอง (No Custom Master Data/Product Creation):** ห้ามทำการเพิ่มหรือสร้างรายการสินค้าใหม่ (Product) หรือข้อมูลหลัก (Master Data) เข้าระบบเองเพื่อวัตถุประสงค์ในการทดสอบอย่างเด็ดขาด เนื่องจากโรงงานมีรายการสินค้าที่ใช้งานจริงอย่างจำกัดและตายตัว หากต้องการทดสอบระบบ ให้เลือกใช้งานเฉพาะรายการข้อมูลสินค้าที่มีอยู่เดิมในระบบอยู่แล้วเท่านั้น
* **การจำกัดสโคปงานอย่างเคร่งครัด (Strict Scope Control):** เอเจนต์ต้องทำเฉพาะงานที่ผู้ใช้ระบุหรือสั่งการโดยตรงเท่านั้น ห้ามดำเนินการนอกเหนือคำสั่งหรือวิเคราะห์ปรับปรุงเองแบบก้าวล่วงสโคปงาน
* **ห้ามเริ่มทำงานจนกว่าจะส่งมอบภารกิจครบ (No Early Execution until All Tasks Assigned):** หากผู้ใช้แจ้งว่ามีรายการภารกิจหลายข้อที่จะทยอยมอบหมาย (เช่น แจ้งว่ามี 5 งานที่จะส่งมาทีละข้อ) ห้ามเอเจนต์เริ่มต้นแก้ไขโค้ด รันคำสั่งแก้ไข หรือดัดแปลงตรรกะระบบใด ๆ ทั้งสิ้นจนกว่าผู้ใช้จะส่งรายการครบถ้วนและมีคำสั่งสั่งการให้เริ่มปฏิบัติงานอย่างชัดเจน โดยเอเจนต์จะทำเพียงแค่การบันทึกรายการงาน (Task list) และแจ้งตอบรับเท่านั้น
* **การจำกัดการทดสอบ UAT ด้วยเบราว์เซอร์ (Browser/DOM UAT Testing Restriction):** หากผู้ใช้สั่งให้ทำการแก้ไขหรือปรับปรุงโค้ดส่วนใด ให้เอเจนต์ทำการแก้ไขโค้ดและรันคอมไพล์ (`type-check` / `lint` / `build`) ในเครื่องเพื่อให้ผ่านอย่างเรียบร้อยเท่านั้น **ห้ามทำการทดสอบผ่านเบราว์เซอร์หรือ DOM Testing ด้วย Browser agent โดยเด็ดขาด เว้นแต่ผู้ใช้จะระบุคำสั่งให้ทำการทดสอบอย่างชัดแจ้ง** (หากเป็นการแก้ไขปรับปรุงเฉยๆ ก็ให้แก้ไขเฉยๆ ไม่ต้องเทส)
* **ห้ามทำ Git Commit / Push เอง:** หากผู้ใช้สั่งให้ปรับแต่งโค้ดหรือแก้ไขส่วนใดก็ตาม ให้ทำเฉพาะการแก้ไขภายในเครื่องพัฒนาแบบโลคอล (Local modifications) เท่านั้น **ห้ามทำ `git commit` หรือ `git push` ด้วยตนเองโดยเด็ดขาด** (การ commit หรือ push จะกระทำก็ต่อเมื่อผู้ใช้เป็นคนพิมพ์สั่งการอย่างชัดเจนเท่านั้น)
* **การส่งและอัปเดตสาขา (Branch Synchronization Rule):** เมื่อทำงานเสร็จสิ้นและได้รับคำสั่งให้ push หรืออัปเดตโค้ด **จะต้องทำการ push ไปที่สาขา `dev` ของ `new-origin` เท่านั้น (ไม่จำเป็นต้อง push ขึ้น uat)** ห้ามทำการ push ไปที่ `origin` หรือ `main` เด็ดขาดเพื่อความปลอดภัยของระบบ UAT และ production
* **ความใส่ใจในรายละเอียดและความสม่ำเสมอของสไตล์ (Design Consistency & Detail Polish):** เมื่อผู้ใช้ล็อกรูปแบบการดีไซน์ที่ต้องการ (เช่น ดีไซน์แบบ AcexPOS ที่ไม่มีกรอบขอบดำเข้ม ไม่มีขีดดำรอบปุ่ม/โฟกัส และใช้เส้นแบ่งแบบบาง Slate-100) เอเจนต์จะต้องปฏิบัติตามมาตรฐานนี้ทุกจุดเมื่อได้รับมอบหมายให้แต่งหน้าจอหรือโมดูลอื่น ๆ โดยต้องแก้ไขอย่างสมบูรณ์แบบ (100% parity) ห้ามทำงานครึ่ง ๆ กลาง ๆ หรือหลงเหลือองค์ประกอบที่ผู้ใช้ไม่ต้องการเด็ดขาด
* **แยกแยะคำถามกับการลงมือทำ (Inquiry vs Action Policy):** เมื่อผู้ใช้สอบถามหรือถามข้อมูลใด ๆ (เช่น ถามว่ามีสกิลนี้ไหม, ถามว่าโค้ดส่วนนี้ทำอะไร) ให้ถือว่าเป็นการถามเพื่อรับข้อมูลเท่านั้น เอเจนต์จะต้องตอบเพียงคำถามและเตรียมรอคำสั่งถัดไป **ห้ามตีความคำถามเพื่อเป็นคำสั่งในการแก้ไขโค้ด ติดตั้งระบบ หรือเปลี่ยนโครงสร้างฐานข้อมูลโดยพลการ** เว้นแต่ผู้ใช้จะระบุคำสั่งให้ทำ (Action) อย่างชัดแจ้ง
* **หยุดถามเมื่อไม่แน่ใจ (Ask When Unsure):** ก่อนที่จะเริ่มทำหรือปรับแต่งโค้ด หากมีจุดใดที่เกิดความลังเล สงสัย หรือสเปกงานมีความคลุมเครือ/มีหลายแนวทางให้เลือก เอเจนต์จะต้องหยุดและใช้งานเครื่องมือถามคำถาม (`ask_question` tool) เพื่อแสดงตัวเลือก Modal ให้ผู้ใช้ยืนยันการตัดสินใจทันที ห้ามวิเคราะห์คิดเดาแนวทางเองโดยพลการ
* **สร้างเช็คลิสต์งานก่อนเริ่มงานเสมอ (Create Task Checklist Before Starting):** ทุกครั้งที่จะเริ่มลงมือทำภารกิจหรือแก้ไขโค้ด เอเจนต์จะต้องจัดทำเช็คลิสต์งาน (Task Checklist) ลงในไฟล์งานประจำวัน (เช่น `task15-06-26.md` หรือ `task.md`) เพื่อให้ลำดับขั้นตอนในการทำงานมีความชัดเจน เป็นระเบียบเรียบร้อย และสามารถตรวจสอบสถานะการดำเนินงานได้ในทุกขั้นตอนก่อนแก้ไขจริง
* **รายงานสั้นกระชับเมื่อจบงาน (Brief Final Reports):** เมื่อทำงานเสร็จสิ้น เอเจนต์จะต้องรายงานสรุปผลลัพธ์อย่างสั้นและกระชับที่สุด หลีกเลี่ยงการอธิบายโค้ดที่ยาวเหยียดเกินความจำเป็น เพื่อประหยัด Token ของเซสชันและเน้นผลลัพธ์ที่จับต้องได้จริง
* **การตรวจสอบก่อนส่งมอบงาน (Validation & Verification):** ทุกครั้งที่ทำการแก้ไข UI หรือโค้ดใด ๆ เอเจนต์ต้องทำความสะอาดและตรวจสอบความถูกต้องผ่านคำสั่งทดสอบ (เช่น `npm run lint` และ `type-check`) บนเครื่องโลคอลก่อนเสมอ เพื่อรับประกันความพร้อมของซอร์สโค้ดและหลีกเลี่ยงการส่งมอบงานที่มีจุดบกพร่อง
* **รักษาตรรกะเดิมและความคิดเห็นที่ไม่เกี่ยวข้อง (Preserve Unrelated Code):** เอเจนต์ต้องอนุรักษ์ตรรกะโค้ด ความคิดเห็นอธิบายโค้ด (Comments) และ Docstrings เดิมทั้งหมดที่ไม่เกี่ยวข้องกับงานที่ได้รับสั่งการ ห้ามลบหรือดัดแปลงโดยพลการเพื่อให้ระบบสืบย้อนกลับและคงคุณภาพสูงสุด
* **แจ้งเตือนปัญหาทันที (Fail-Fast Policy):** หากพบปัญหาการชนกันของข้อมูล (Data Conflict), ความไม่เข้ากันของประเภทข้อมูล (Type Mismatch) หรือคอมไพล์ไม่ผ่านระหว่างทำงาน เอเจนต์ต้องหยุดงานและแจ้งให้ผู้ใช้ทราบทันทีเพื่อขอคำชี้แนะ ห้ามพยายามแก้ไขข้อมูลหรือตรรกะในส่วนสำคัญด้วยตนเองโดยไม่ได้ปรึกษา
* **การส่งมอบงานด้วยภาพหลักฐาน (Visual Verification):** ทุกครั้งที่ทำการแก้ไขหรือปรับปรุง UI หน้าจอเสร็จสิ้น เอเจนต์ต้องบันทึกรูปภาพ (Screenshot) หรือวิดีโอแสดงผลลัพธ์มาแนบในรายงานสรุปเสมอ เพื่อให้ผู้ใช้สามารถตรวจความถูกต้องทางสายตาได้อย่างรวดเร็ว
* **รับประกันสัดส่วนหน้าจอมือถือ (Mobile Responsiveness Guarantee):** การปรับแต่ง UI จะต้องทดสอบและตรวจสอบการแสดงผลบนขนาดหน้าจอขนาดเล็ก 375px (Mobile viewport) เสมอ เพื่อรับประกันว่าไม่มีส่วนของเนื้อหาหรือปุ่มตกขอบ ล้น หรือบิดเบี้ยว
* **รายงานสรุปประวัติ Commit (Commit Summary Policy):** เมื่อได้รับคำสั่งให้ทำ Git Commit หรือ Push งาน เอเจนต์จะต้องระบุข้อความ Commit (Commit Message) และรหัส Commit Hash ล่าสุดอย่างย่อมาในรายงานส่งท้ายเสมอ เพื่อความง่ายในการอ้างอิงรุ่นซอร์สโค้ด
* **วิเคราะห์ภาพและตรวจเช็คความละเอียดของ UI (Visual Image Feedback & Audit Policy):** เมื่อผู้ใช้ส่งภาพหน้าจอตัวอย่างหรือจุดแก้ไขมาให้ เอเจนต์ต้องวิเคราะห์ภาพอย่างละเอียด ตรวจสอบ Responsive และองค์ประกอบต่าง ๆ ทั้งหมด หากพบสิ่งผิดปกติ เช่น เส้นขีดดำเข้มที่หลงเหลืออยู่ หรือข้อความล้นกรอบแสดงผลไม่สวยงาม จะต้องดำเนินการแก้ไขข้อบกพร่องทางสายตาเหล่านี้ให้สมบูรณ์แบบก่อนส่งมอบรายงานสรุปผลลัพธ์
* **หลีกเลี่ยงตารางที่ต้องเลื่อนขวาบน Mobile (No Horizontal Scroll Tables on Mobile):** บนอุปกรณ์หน้าจอขนาดเล็ก (Mobile/Tablet) ห้ามใช้การแสดงผลแบบตารางที่ต้องเลื่อนแถบด้านขวา (Horizontal Scroll) เป็นอันขาด เพราะทำให้ใช้งานยากและไม่เป็นระเบียบ ให้เปลี่ยนไปแสดงผลด้วยโครงสร้างการ์ดแนวตั้ง (Mobile Dense Card List) ที่มีความเป็นระเบียบ สวยงาม และอ่านง่ายที่สุดแทน
* **การทดสอบความถูกต้องบนเบราว์เซอร์อย่างละเอียด (Thorough Interactive Browser Testing):** เมื่อได้รับมอบหมายให้ตรวจสอบหรือทดสอบระบบ เอเจนต์ต้องเรียกใช้งานเครื่องมือควบคุมเบราว์เซอร์เพื่อเปิดดูโครงสร้าง DOM ลองกรอกข้อมูลในฟอร์มให้ครบถ้วนเพื่อจำลองการบันทึกรายการ/สร้างเอกสารจริง และทดสอบฟังก์ชันใช้งานทั้งหมด หากพบปุ่มหรือหน้าจอส่วนใดที่สามารถคลิกใช้งานได้หรือไม่ได้ และด้วยเหตุผลใด เอเจนต์จะต้องรายงานสาเหตุรวมถึงอธิบายแนวทางแก้ไขให้ผู้ใช้ทราบอย่างชัดเจน **นอกจากนี้ ห้ามข้ามการทดสอบหรือเปิดโมดอลแล้วกดปิดเฉย ๆ แต่จะต้องทำจนจบกระบวนการจริง (End-to-End) เช่น การทดสอบและกดสร้างเอกสารบิลรับซื้อทั้งโหมด STOCK และโหมด TRADING ให้สำเร็จจริงในระบบ เพื่อยืนยันว่าการแก้ไขไม่กระทบต่อฟังก์ชันการทำงานหลัก**
* **การวิเคราะห์ข้อมูลและทดสอบอย่างมีประสิทธิภาพ (Browser Testing Efficiency & Pre-Analysis Rule):** ก่อนเริ่มเรียกใช้งาน Browser Agent หรือ UAT เบราว์เซอร์ เอเจนต์หลักจะต้องวิเคราะห์และสืบค้นข้อมูลล่วงหน้าอย่างละเอียด เช็คข้อมูลประเภทฟิลด์ ข้อกำหนดความเข้ากันได้ของชนิดข้อมูลจากโค้ดหลังบ้าน และข้อมูลจริงในฐานข้อมูล (เช่น รายการสินค้าที่มีอยู่จริง, ยอดสต๊อกคงเหลือ และความถูกต้องของชนิดข้อมูล) ก่อนที่จะรันสคริปต์กรอกหรือคลิกในเบราว์เซอร์ เพื่อให้แก้ไขปัญหาและผ่าน UAT สำเร็จลุล่วงได้ภายใน **2-3 ครั้งของการทดสอบ** หลีกเลี่ยงการปล่อยให้บอทเบราว์เซอร์กดทดสอบ คาดเดาข้อมูล หรือวนซ้ำจุดผิดพลาดเดิมซ้ำซากโดยไม่แก้ปัญหาเชิงวิเคราะห์
* **กฎการหยุดและสืบค้นเมื่อเกิดข้อผิดพลาดในการทำงานของ DOM (DOM Failure Halt & Research Rule):** ตอนใช้ DOM ถ้าผิดก็กลับมาคิดวิเคราะห์ค้นหาก่อน! ห้ามปล่อยให้บอทหรือ subagent ทำการสุ่มเดาข้อมูล ลองเดาค่า ลองคลิก หรือรันกระบวนการวนซ้ำๆ เพื่อลองผิดลองถูกในหน้าจอโดยเด็ดขาด หากเกิดข้อผิดพลาดหรือติดขัดในขั้นตอนใดๆ ของการทำงานและการทดสอบผ่าน DOM/เบราว์เซอร์ (ครอบคลุมทุกการทำงาน ไม่ใช่เฉพาะการ Login เช่น การคลิกปุ่มหาไม่เจอ, การกรอกฟิลด์ข้อมูลผิดพลาด, การเลือกตัวกรอง หรือการทำรายการธุรกรรมต่างๆ) เอเจนต์ต้องหยุดการทำงานทันที แล้วกลับมาคิดวิเคราะห์ ค้นหาข้อมูลข้อกำหนด (API payload, config, schema, seed data, หรือ logic ในโค้ด) ในเครื่องก่อนเสมอ เพื่อทำความเข้าใจให้ถูกต้องก่อนจะดำเนินการต่อ
* **การจำกัดพื้นที่การทดสอบตามคำสั่ง (Scoped Testing Policy):** การทดสอบระบบจะต้องจำกัดพื้นที่อยู่เฉพาะส่วน/หน้าจอที่ได้รับมอบหมายเท่านั้น (เช่น หากให้ทดสอบบิลรับซื้อ ให้เจาะลึกเฉพาะฟังก์ชันในหน้าจอระบบบิลรับซื้อ) ห้ามออกไปกดหรือทดสอบระบบโมดูลอื่นนอกเหนือจากที่ระบุในสโคปงาน เว้นแต่จะได้รับคำสั่งให้ออดิตทั้งโมดูลหรือทั้งแอปพลิเคชัน
* **การตรวจสอบส่วนประกอบปุ่มและตัวกรองบางส่วน (Tabs, Filters, and Table Click Testing):** บนหน้าจอที่ได้รับมอบหมาย ปุ่มและฟังก์ชันการกรองที่เกี่ยวข้องจะต้องถูกทดสอบการคลิกเพื่อตรวจสอบการทำงาน โดยครอบคลุม:
  - **แท็บส่วนหัวหน้าจอ (Header Tabs):** เช่น แท็บ "บิลรับซื้อ" และแท็บ "ประวัติเปลี่ยนตัว Supplier"
  - **ปุ่มตัวกรอง (Filter Options):** สำหรับตัวกรองประเภทและสถานะ ให้ทดลองกดสุ่มเลือกเพียงบางปุ่มพอเป็นตัวอย่าง (เช่น เลือกประเภท 1 เงื่อนไข และเลือกสถานะ 1 เงื่อนไข) เพื่อยืนยันว่าหน้าจอมีการคัดกรองข้อมูลถูกต้องและเรียกใช้ API ได้จริง ไม่จำเป็นต้องไล่กดสลับทดสอบจนครบทุกปุ่มทุกตัวเพื่อเป็นการประหยัดเวลาและทรัพยากรในการตรวจสอบ
  - **การคลิกเปิดดูรายละเอียดในตาราง (Table Detail & Row Clicks):** ทดลองคลิกแถวในตารางเพื่อดูว่ากล่องเปิดแสดงรายละเอียด (Detail Modal) แสดงข้อมูลขึ้นมาถูกต้อง อ่านง่าย สวยงามเป็นระเบียบหรือไม่
* **การตรวจสอบแบบ Responsive ในโหมดพกพาหลังเดสก์ท็อป (Mobile/Responsive Verification after Desktop):** หลังจากทดสอบในโหมดเดสก์ท็อปเรียบร้อยแล้ว จะต้องจำลองการแสดงผลและทดสอบปุ่ม ตัวกรอง ตารางข้อมูลในขนาดหน้าจอมือถือ (เช่น 375px) เสมอ เพื่อให้มั่นใจว่าปุ่มกดได้ง่าย ตารางถูกแปลงเป็น Card list สวยงามไม่มีการเลื่อนด้านขวา (Horizontal scroll) และข้อมูลทั้งหมดแสดงผลจัดระเบียบเรียบร้อยในสัดส่วนหน้าจอนั้น ๆ
* **การวิเคราะห์จุดบกพร่องและเสนอแนะเพื่อปรับปรุงทางสายตา (Proactive Visual Self-Audit & "Eh" Policy):** ในการรันเบราว์เซอร์และอ่านข้อมูล DOM เอเจนต์ต้องตั้งใจสังเกตและคิดวิเคราะห์ (เอ๊ะ) เพื่อตรวจหาจุดบกพร่องทางสายตา เช่น ข้อความซ้อนทับกัน ตารางที่บีบเกินไปจนอ่านยาก หรือปุ่มที่เว้นระยะไม่สมส่วน หากพบจุดเหล่านี้ ให้จดบันทึก สรุปรายละเอียด และเสนอแนะแนวทางแก้ไข (เช่น การปรับเป็นตาราง auto-fit หรือการขยับ spacing) ให้ผู้ใช้พิจารณาตัดสินใจก่อนลงมือแก้ไขจริงเสมอ เพื่อควบคุมคุณภาพ UI ให้เป็นไปตามสไตล์ความชอบของผู้ใช้และมาตรฐานระบบอย่างสูงสุด
* **บันทึกสรุป Flow การทำงานของระบบหลังการแก้ไขเสร็จสิ้นทั้งหมด (Final Flow Summary Requirement):** กฎการทำงานสำคัญที่กำหนดโดย USER คือ เอเจนต์จะต้องบันทึกสรุป Flow การทำงานของโมดูลหรือฟังก์ชันที่มีการแก้ไข หลังจากพัฒนาหรือแก้ไขงานนั้นๆ เสร็จสิ้นเรียบร้อยแล้วทุกครั้ง (โดยการสรุปและอัปเดตลงในเอกสาร Flow อ้างอิงทางธุรกิจของโปรเจกต์ที่เกี่ยวข้องโดยเฉพาะ เช่น เอกสารใน `docs/notes/` หรือไฟล์ Flow หลัก) ห้ามข้ามขั้นตอนนี้เด็ดขาด เพื่อให้ทีมงานสามารถติดตามการเปลี่ยนแปลงและตรวจสอบ Flow การทำงานหลังแก้ไขได้ครบถ้วน
* **การซิงค์รหัสล่าสุดก่อนเริ่มภารกิจใหม่ (Upstream Sync Rule):** ก่อนเริ่มต้นเปิดสเปกแก้ไขงานหรือเริ่ม Task ใหม่ของวัน เอเจนต์จะต้องทำการอัปเดตประวัติ Git ล่าสุดจาก Remote `new-origin/dev` เสมอ เพื่อหลีกเลี่ยงปัญหาโค้ดในเครื่องโลคอลล้าสมัยกว่าบนเซิร์ฟเวอร์หลัก (ป้องกันการเกิด Merge Conflicts ที่แก้ไขยากในภายหลัง)
* **การรักษาความสะอาดของ Workspace (Staging/Workspace Cleanliness Rule):** ก่อนเสนอให้ผู้ใช้อนุมัติทำ Git Commit หรือ Push ทุกครั้ง เอเจนต์ต้องรัน `git status` เพื่อล้าง/แยกไฟล์ขยะจากการรันโปรแกรม หรือ log ไฟล์ชั่วคราวออกไปให้หมด และต้อง Add เฉพาะไฟล์โค้ดกับเอกสารที่เกี่ยวข้องกับภารกิจนั้นจริง ๆ เท่านั้น
* **การคอมไพล์แบบทีละขั้นตอน (Incremental Compilation Rule):** หากภารกิจหนึ่งจำเป็นต้องแก้ไขไฟล์โค้ดหลายไฟล์ เอเจนต์จะต้องทำตามกฎ "รัน `npm run type-check` และ `lint` ทันทีหลังแก้ไขแต่ละจุดหรือแต่ละไฟล์เสร็จสิ้น" ก่อนจะข้ามไปแก้ไขไฟล์ตัวถัดไป เพื่อช่วยให้ตรวจพบข้อผิดพลาดด้าน Type หรือ Syntax ได้ทันที ไม่ปล่อยสะสมจนเกิดการพังแบบทับซ้อนกันหลายไฟล์
* **ความปลอดภัยทางข้อมูลระดับธุรกรรม (Database Write Safety Rule):** ในการทดสอบแก้ไขหรือส่งข้อมูลผ่าน API ที่มีผลต่อฐานข้อมูลใน Supabase target dev:
  - หากมีการเขียนสคริปต์ตรวจสอบ (Contract Verification Script) จะต้องทำงานภายใต้ **Prisma Transaction ที่มีการเรียก Rollback เสมอ** ในขั้นตอนทดสอบ (ห้ามทำการ Commit จริงในสคริปต์ทดสอบโดยเด็ดขาด)
  - ข้อมูลจริงใน Database จะถูกเขียนลงไปก็ต่อเมื่อเป็นการรันระบบบนเบราว์เซอร์เพื่อทำการทดสอบ UAT ด้วยมือแบบ End-to-End เท่านั้น
* **การส่งต่องานระหว่างเซสชันอย่างเป็นทางการ (Resumable Session Handoff Rule):** ในช่วงท้ายของแต่ละรอบการทำงาน เอเจนต์จะต้องจัดเตรียมข้อความสรุปการดำเนินงานล่าสุด ปัญหาที่ค้างคา และขั้นตอนถัดไป (Next Steps) สำหรับบันทึกเก็บไว้ในตัวติดตาม เพื่อให้เอเจนต์คนถัดไปสามารถมารับช่วงต่อและทำงานได้อย่างไร้รอยต่อโดยไม่ต้องอธิบายข้อมูลใหม่ทั้งหมด
* **การป้องกันการแก้ไขไฟล์ทับซ้อนกัน (File-level Isolation Rule):** เพื่อหลีกเลี่ยงข้อขัดแย้งของโค้ดและการทับซ้อนข้อมูลกับผู้พัฒนาอื่น (Devs) เอเจนต์จะต้องไม่แก้ไขไฟล์ที่มีผู้พัฒนารายอื่นกำลังทำงานอยู่พร้อมกันในขณะนั้น หากผู้ใช้แจ้งว่าไฟล์เป้าหมายมีผู้พัฒนาท่านอื่นแก้ไขอยู่ เอเจนต์จะต้องหยุดงานชั่วคราวหรือเปลี่ยนไปพัฒนาฟีเจอร์/โมดูลในไฟล์อื่นที่ไม่มีการทับซ้อนกันแทน

เพื่อช่วยให้การปรับปรุงและพัฒนาระบบในโมดูลถัดไปมีความเป็นระเบียบ แข็งแกร่ง และปลอดภัยสูงสุด ได้กำหนดข้อแนะนำการปฏิบัติทางเทคนิคเพิ่มเติมดังนี้:

   - เหตุผล: เพื่อให้ SearchCombobox ตรวจหาพิกัด Portal และเรนเดอร์ dropdown panel ภายใต้ Dialog DOM tree ได้โดยอัตโนมัติ ช่วยป้องกันปัญหา Focus Trap และ Pointer Block ของ Radix UI ที่ทำให้ผู้ใช้กดเลือกสินค้าไม่ได้บนเบราว์เซอร์จริง
2. **การทำ Input Validation ด้วย Zod Schema ในฟอร์มทำรายการ:**
   - สำหรับการป้อนข้อมูลในอนาคต แนะนำให้ปรับเปลี่ยนการใช้ Manual validation (เขียนเงื่อนไขเช็คข้อมูลด้วยมือแบบ inline) มาใช้การตรวจสอบข้อมูลผ่าน **Zod Schema** เสมอ
   - เหตุผล: ช่วยเพิ่มความปลอดภัยของข้อมูลในระดับ Type-safety ป้องกันปัญหา Malformed data หลุดรอดเข้าไปยัง API payload และง่ายต่อการปรับแต่ง/จัดระเบียบหน้าจอแสดงข้อความผิดพลาด (Error messages)
3. **การทำ Database Seeding จากข้อมูลสินค้าจริงของโรงงาน (Real-data Mocking):**
   - เพื่อให้สอดคล้องกับข้อกำหนดการห้ามสร้างข้อมูลสินค้าขึ้นมาเองในการทดสอบ แนะนำให้ทำการดึงชุดข้อมูลสินค้าหลัก (Master Product List) ล่าสุดจากโรงงานจริง มาตั้งค่าเป็นฐานข้อมูลจำลอง (Dev Database Seeding) โดยตรง
   - เหตุผล: เพื่อช่วยให้การพัฒนาและทดสอบระบบทั้งหมดดำเนินไปภายใต้รายการข้อมูลสินค้าที่มีอยู่จริงของโรงงานเท่านั้น ลดโอกาสการปะปนของข้อมูลขยะและช่วยทดสอบความถูกต้องของระบบคำนวณบัญชีคลังสินค้าได้อย่างแม่นยำที่สุด

## 🧪 ผลการทดสอบ UAT และข้อสังเกตโมดูลการผลิต (UAT Audit Results & Bug Observations - 2026-06-15)

จากการรัน UAT และตรวจสอบการเคลื่อนไหวของคลังสินค้าและบัญชีคลังฝ่ายผลิต ผ่านเบราว์เซอร์และสคริปต์ มีข้อสังเกตและพฤติกรรมของระบบที่ต้องบันทึกดังนี้:

### 1. พฤติกรรมการคำนวณและอัปเดตสถานะ (Correct Behaviors)
- **Real-time Metrics:** บนหน้ารายละเอียดใบสั่งผลิต ยอดวัตถุดิบนำเข้า (`inputQty`), ผลผลิตรวม (`outputQty`), ยอดสูญเสีย (`lossQty`), WIP คงเหลือ และ Yield % คำนวณแบบสะสมจากรายการเคลื่อนไหวได้ถูกต้องตามจริง
- **Status Transition:** เอกสารมีวงจรสถานะที่สมบูรณ์ `Open -> In Production -> Partially Completed -> Completed` (หากยอด WIP balance = 0 และระบุจบงาน) หรือเปลี่ยนเป็น `Cancelled` ได้สำเร็จ
- **Lock Rules:** เมื่อใบสั่งผลิตอยู่ในสถานะ `Completed` หรือ `Cancelled` ระบบจะบล็อกการบันทึก Input/Output เพิ่มเติม และปุ่มย้อนกลับรายการ (Reverse) จะถูก Disable เสมอ ซึ่งตรงตาม Business Rules

### 2. บั๊กที่ตรวจพบในหน้าจอการผลิต (Bug Observations)
- **บั๊ก Case-sensitivity ของสถานะการเคลื่อนไหว (`active` vs `Active`):**
  - **ไฟล์ที่พบปัญหา:** [ProductionOrdersPageClient.tsx](file:///c:/new-ns-scrap-erp/apps/next/src/components/production/ProductionOrdersPageClient.tsx)
  - **รายละเอียด:** โค้ดส่วน UI ตรวจสอบสถานะเพื่อประเมินปุ่มย้อนกลับรายการโดยเช็คกับเงื่อนไข `row.status !== 'Active'` (ตัวพิมพ์ใหญ่) และ Badge ตรวจสอบ `row.status === 'Active'` แต่ทว่าข้อมูล JSON จริงจาก API/Database ส่งค่าสถานะมาเป็น `"active"` (ตัวพิมพ์เล็ก)
  - **ผลกระทบ:** 
    1. ปุ่ม **Reverse** สำหรับเบิกวัตถุดิบ (Input) และรับผลผลิต (Output) ถูก Disable เสมอ ทำให้ผู้ใช้ทั่วไปไม่สามารถกด Reverse รายการได้แม้สถานะใบสั่งผลิตจะเอื้ออำนวย
    2. สี Badge สถานะแสดงผลเป็นสีแดง (เสมือนว่ารายการผิดปกติ) แทนที่จะเป็นสีเขียว
  - **แนวทางแก้ไข:** เปลี่ยนการตรวจสอบเงื่อนไขใน [ProductionOrdersPageClient.tsx](file:///c:/new-ns-scrap-erp/apps/next/src/components/production/ProductionOrdersPageClient.tsx) ให้เทียบค่ากับ `'active'` (หรือให้ Tolerate ทั้งสองกรณีด้วยการใช้ `.toLowerCase()`) เพื่อให้สอดคล้องกับ API payload จริง
