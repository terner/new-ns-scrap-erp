# Peach UI Guidelines & Reference (AcexPOS Style)

เอกสารนี้รวบรวมข้อกำหนดการออกแบบและการปรับปรุงระบบอินเตอร์เฟส (UI Rehabilitation) ที่เกิดขึ้นในบรันช์ `peach` (Commit: `7223e7a` เมื่อ 2026-06-13) เพื่อใช้เป็นแนวทางมาตรฐานในการพัฒนาหน้าจอการแสดงผลอื่น ๆ ของระบบ **NS Scrap ERP** ให้มีลักษณะเดียวกับสไตล์ **AcexPOS**

---

## 🎨 หลักการดีไซน์สำคัญ (AcexPOS UI Standard)

### 1. โครงสร้างการ์ดข้อมูลและตัวชี้วัด (KPI / Metric Summary Cards)
สำหรับกล่องตัวเลขสถิติหรือสรุปสถานะด้านบนของหน้าเว็บ (KPI Overview):
- **กล่องขอบนอก (Outer Wrapper):** ใช้สีพื้นหลังและสีขอบเป็น `bg-slate-50 border-slate-200`
- **การ์ดข้อมูลภายใน (Inner Cards):**
  - สไตล์: `bg-white shadow-sm border border-slate-200 rounded-xl`
  - ตกแต่งไอคอนด้านซ้าย: ใช้ไอคอนพร้อมพื้นหลังวงกลมสีพาสเทลอ่อน (Circular Pastel Background) เพื่อให้ดูพรีเมียมและสวยงาม
  - ตัวเลขเด่นชัด และมีชื่อกำกับชัดเจนอยู่ทางด้านขวา
  - **🚫 ข้อห้ามสำคัญ:** ห้ามใช้รูปแบบ Left-border indicator / Left-border Accent (การ์ดที่มีแถบสีขอบด้านซ้ายหนา) โดยเด็ดขาด เพราะทำให้ขาดความเป็นระเบียบและไม่สวยงาม
  - **⚖️ ตรรกะสีตัวเลขทางการเงิน:** ในกรณีการ์ดที่แสดงผลต่างหรือ Margin หากค่าตัวเลขเป็นศูนย์ (`0.00` หรือไม่มีส่วนต่าง) ต้องแสดงผลในโทนสีเทาที่เป็นกลาง (`Neutral Slate` เช่น ไอคอนใช้ `bg-slate-100 text-slate-600` และตัวเลขใช้ `text-slate-900`) ห้ามใช้สีเขียวหรือแดงพร่ำเพรื่อเพื่อป้องกันการตีความคลาดเคลื่อนทางการเงิน

### 2. การจัดวางแถวตัวกรองและปุ่มปฎิบัติการ (Toolbar / Filters layout)
- **การปัดขวาของปุ่มหลัก (Desktop Actions):** ปุ่มคำสั่งหลัก (เช่น Refresh, Export Excel, หรือ เพิ่มรายการใหม่) บนหน้าจอ Desktop ให้ผลักไปอยู่มุมบนขวาของแถวตัวกรอง (Filter Row) โดยใช้คลาส Tailwind `ml-auto`
- **การตอบรับบน Mobile:**
  - ใช้ **Filters Drawer** หรือ **Bottom Sheet** สำหรับหน้าจอขนาดเล็กเพื่อซ่อนตัวกรองที่เยอะเกินไป
  - แยก Toolbar ระหว่าง Desktop และ Mobile ให้มีสไตล์เฉพาะตัว

### 3. ตารางข้อมูลหลัก (Data Table & Grid)
- **การยุบตารางบนมือถือ:** ตารางขนาดใหญ่จะถูกซ่อนบนหน้าจอขนาดเล็ก (`hidden md:table`) และแทนที่ด้วยรายการการ์ดกระชับ (Mobile Dense Card List) ที่สามารถเลื่อนอ่านได้สะดวก
- **เส้นขอบตารางที่นุ่มนวล:** ห้ามใช้เส้นตารางขอบขีดดำเข้มของเบราว์เซอร์ดีฟอลต์ ให้เปลี่ยนมาใช้เส้นตารางสีนุ่มนวล เช่น `border-slate-100` หรือ `border-slate-200/60`
- **ลบ Focus Outline:** ปิดแถบขอบดำที่เกิดจากการกดเลือกของเบราว์เซอร์ดีฟอลต์ (Focus Outline) ในส่วนต่าง ๆ เช่น ปุ่ม และอินพุต โดยการเพิ่มคลาส `outline-none` หรือการกำหนดสไตล์โกลบอลใน `globals.css`

### 4. ป๊อปอัปและฟอร์มทำรายการ (Modals / Forms Layout)
- **หัวข้อบาร์แบบเข้ม (Dark Header):** ฟอร์มหลักและกล่องรายละเอียดข้อมูล (Modal/Dialog) ให้ใช้แถบหัวเรื่องเป็นสีเข้ม `bg-slate-900` ร่วมกับข้อความสีขาว `text-white`
- **ส่วนท้ายฟอร์ม (Form Footer):**
  - ปุ่มยกเลิกและปุ่มยืนยันให้จัดชิดขวาร่วมกัน
  - ปุ่มยกเลิกใช้รูปแบบข้อความเรียบง่ายไร้กรอบ (Text-only Button) และปุ่มตกลง/บันทึกใช้ดีไซน์หลัก เช่น สีเด่นมีน้ำหนักมนสวยงาม (เช่น `bg-[#0F172A] hover:bg-[#1E293B]`)

## ⚠️ ข้อควรระวังเพิ่มเติมในการพัฒนา UI
* **ห้ามมีกรอบขอบดำรอบปุ่มหรืออินพุตเมื่อถูกเลือก:** ตรวจสอบให้มั่นใจว่าปุ่มใช้งานไม่มี Browser Focus Ring สีดำหนา
* **การออกแบบ Modal:** ต้องครอบคลุมพฤติกรรมบนอุปกรณ์เคลื่อนที่ด้วยเสมอ โดยเฉพาะเรื่องความสูงของหน้าจอและระยะสัมผัส (Touch Targets)
* **การจัดการรูปสินค้า:** ให้ใช้มาตรฐาน Storage bucket `product-images` ในการจัดเก็บรูปภาพสินค้า (อ้างอิงจาก Standard ใหม่ในระบบคลัง)

## 🚫 กฎควบคุมการทำงาน (Git & Scope Rules)
* **การจำกัดสโคปงานอย่างเคร่งครัด (Strict Scope Control):** เอเจนต์ต้องทำเฉพาะงานที่ผู้ใช้ระบุหรือสั่งการโดยตรงเท่านั้น ห้ามดำเนินการนอกเหนือคำสั่งหรือวิเคราะห์ปรับปรุงเองแบบก้าวล่วงสโคปงาน
* **ห้ามทำ Git Commit / Push เอง:** หากผู้ใช้สั่งให้ปรับแต่งโค้ดหรือแก้ไขส่วนใดก็ตาม ให้ทำเฉพาะการแก้ไขภายในเครื่องพัฒนาแบบโลคอล (Local modifications) เท่านั้น **ห้ามทำ `git commit` หรือ `git push` ด้วยตนเองโดยเด็ดขาด** (การ commit หรือ push จะกระทำก็ต่อเมื่อผู้ใช้เป็นคนพิมพ์สั่งการอย่างชัดเจนเท่านั้น)
* **ความใส่ใจในรายละเอียดและความสม่ำเสมอของสไตล์ (Design Consistency & Detail Polish):** เมื่อผู้ใช้ล็อกรูปแบบการดีไซน์ที่ต้องการ (เช่น ดีไซน์แบบ AcexPOS ที่ไม่มีกรอบขอบดำเข้ม ไม่มีขีดดำรอบปุ่ม/โฟกัส และใช้เส้นแบ่งแบบบาง Slate-100) เอเจนต์จะต้องปฏิบัติตามมาตรฐานนี้ทุกจุดเมื่อได้รับมอบหมายให้แต่งหน้าจอหรือโมดูลอื่น ๆ โดยต้องแก้ไขอย่างสมบูรณ์แบบ (100% parity) ห้ามทำงานครึ่ง ๆ กลาง ๆ หรือหลงเหลือองค์ประกอบที่ผู้ใช้ไม่ต้องการเด็ดขาด
* **แยกแยะคำถามกับการลงมือทำ (Inquiry vs Action Policy):** เมื่อผู้ใช้สอบถามหรือถามข้อมูลใด ๆ (เช่น ถามว่ามีสกิลนี้ไหม, ถามว่าโค้ดส่วนนี้ทำอะไร) ให้ถือว่าเป็นการถามเพื่อรับข้อมูลเท่านั้น เอเจนต์จะต้องตอบเพียงคำถามและเตรียมรอคำสั่งถัดไป **ห้ามตีความคำถามเพื่อเป็นคำสั่งในการแก้ไขโค้ด ติดตั้งระบบ หรือเปลี่ยนโครงสร้างฐานข้อมูลโดยพลการ** เว้นแต่ผู้ใช้จะระบุคำสั่งให้ทำ (Action) อย่างชัดแจ้ง
* **หยุดถามเมื่อไม่แน่ใจ (Ask When Unsure):** ก่อนที่จะเริ่มทำหรือปรับแต่งโค้ด หากมีจุดใดที่เกิดความลังเล สงสัย หรือสเปกงานมีความคลุมเครือ/มีหลายแนวทางให้เลือก เอเจนต์จะต้องหยุดและใช้งานเครื่องมือถามคำถาม (`ask_question` tool) เพื่อแสดงตัวเลือก Modal ให้ผู้ใช้ยืนยันการตัดสินใจทันที ห้ามวิเคราะห์คิดเดาแนวทางเองโดยพลการ
* **สร้างเช็คลิสต์งานก่อนเริ่มงานเสมอ (Create Task Checklist Before Starting):** ทุกครั้งที่จะเริ่มลงมือทำภารกิจหรือแก้ไขโค้ด เอเจนต์จะต้องจัดทำเช็คลิสต์งาน (Task Checklist) ลงในไฟล์งานประจำวัน (เช่น `task15-06-26.md` หรือ `task.md`) เพื่อให้ลำดับขั้นตอนในการทำงานมีความชัดเจน เป็นระเบียบเรียบร้อย และสามารถตรวจสอบสถานะการดำเนินงานได้ในทุกขั้นตอนก่อนแก้ไขจริง
* **รายงานสั้นกระชับเมื่อจบงาน (Brief Final Reports):** เมื่อทำงานเสร็จสิ้น เอเจนต์จะต้องรายงานสรุปผลลัพธ์อย่างสั้นและกระชับที่สุด หลีกเลี่ยงการอธิบายโค้ดที่ยาวเหยียดเกินความจำเป็น เพื่อประหยัด Token ของเซสชันและเน้นผลลัพธ์ที่จับต้องได้จริง
* **การตรวจสอบก่อนส่งมอบงาน (Validation & Verification):** ทุกครั้งที่ทำการแก้ไข UI หรือโค้ดใด ๆ เอเจนต์ต้องทำความสะอาดและตรวจสอบความถูกต้องผ่านคำสั่งทดสอบ (เช่น `npm run lint` และ `type-check`) บนเครื่องโลคอลก่อนเสมอ เพื่อรับประกันความพร้อมของซอร์สโค้ดและหลีกเลี่ยงการส่งมอบงานที่มีจุดบกพร่อง
* **รักษาตรรกะเดิมและความคิดเห็นที่ไม่เกี่ยวข้อง (Preserve Unrelated Code):** เอเจนต์ต้องอนุรักษ์ตรรกะโค้ด ความคิดเห็นอธิบายโค้ด (Comments) และ Docstrings เดิมทั้งหมดที่ไม่เกี่ยวข้องกับงานที่ได้รับสั่งการ ห้ามลบหรือดัดแปลงโดยพลการเพื่อให้ระบบสืบย้อนกลับและคงคุณภาพสูงสุด
* **แจ้งเตือนปัญหาทันที (Fail-Fast Policy):** หากพบปัญหาการชนกันของข้อมูล (Data Conflict), ความไม่เข้ากันของประเภทข้อมูล (Type Mismatch) หรือคอมไพล์ไม่ผ่านระหว่างทำงาน เอเจนต์ต้องหยุดงานและแจ้งให้ผู้ใช้ทราบทันทีเพื่อขอคำชี้แนะ ห้ามพยายามแก้ไขข้อมูลหรือตรรกะในส่วนสำคัญด้วยตนเองโดยไม่ได้ปรึกษา
* **การส่งมอบงานด้วยภาพหลักฐาน (Visual Verification):** ทุกครั้งที่ทำการแก้ไขหรือปรับปรุง UI หน้าจอเสร็จสิ้น เอเจนต์ต้องบันทึกรูปภาพ (Screenshot) หรือวิดีโอแสดงผลลัพธ์มาแนบในรายงานสรุปเสมอ เพื่อให้ผู้ใช้สามารถตรวจความถูกต้องทางสายตาได้อย่างรวดเร็ว
* **รับประกันสัดส่วนหน้าจอมือถือ (Mobile Responsiveness Guarantee):** การปรับแต่ง UI จะต้องทดสอบและตรวจสอบการแสดงผลบนขนาดหน้าจอขนาดเล็ก 375px (Mobile viewport) เสมอ เพื่อรับประกันว่าไม่มีส่วนของเนื้อหาหรือปุ่มตกขอบ ล้น หรือบิดเบี้ยว
* **รายงานสรุปประวัติ Commit (Commit Summary Policy):** เมื่อได้รับคำสั่งให้ทำ Git Commit หรือ Push งาน เอเจนต์จะต้องระบุข้อความ Commit (Commit Message) และรหัส Commit Hash ล่าสุดอย่างย่อมาในรายงานส่งท้ายเสมอ เพื่อความง่ายในการอ้างอิงรุ่นซอร์สโค้ด
* **วิเคราะห์ภาพและตรวจเช็คความละเอียดของ UI (Visual Image Feedback & Audit Policy):** เมื่อผู้ใช้ส่งภาพหน้าจอตัวอย่างหรือจุดแก้ไขมาให้ เอเจนต์ต้องวิเคราะห์ภาพอย่างละเอียด ตรวจสอบ Responsive และองค์ประกอบต่าง ๆ ทั้งหมด หากพบสิ่งผิดปกติ เช่น เส้นขีดดำเข้มที่หลงเหลืออยู่ หรือข้อความล้นกรอบแสดงผลไม่สวยงาม จะต้องดำเนินการแก้ไขข้อบกพร่องทางสายตาเหล่านี้ให้สมบูรณ์แบบก่อนส่งมอบรายงานสรุปผลลัพธ์
* **หลีกเลี่ยงตารางที่ต้องเลื่อนขวาบน Mobile (No Horizontal Scroll Tables on Mobile):** บนอุปกรณ์หน้าจอขนาดเล็ก (Mobile/Tablet) ห้ามใช้การแสดงผลแบบตารางที่ต้องเลื่อนแถบด้านขวา (Horizontal Scroll) เป็นอันขาด เพราะทำให้ใช้งานยากและไม่เป็นระเบียบ ให้เปลี่ยนไปแสดงผลด้วยโครงสร้างการ์ดแนวตั้ง (Mobile Dense Card List) ที่มีความเป็นระเบียบ สวยงาม และอ่านง่ายที่สุดแทน
* **การทดสอบความถูกต้องบนเบราว์เซอร์อย่างละเอียด (Thorough Interactive Browser Testing):** เมื่อได้รับมอบหมายให้ตรวจสอบหรือทดสอบระบบ เอเจนต์ต้องเรียกใช้งานเครื่องมือควบคุมเบราว์เซอร์เพื่อเปิดดูโครงสร้าง DOM ลองกรอกข้อมูลในฟอร์มให้ครบถ้วนเพื่อจำลองการบันทึกรายการ/สร้างเอกสารจริง และทดสอบฟังก์ชันใช้งานทั้งหมด หากพบปุ่มหรือหน้าจอส่วนใดที่สามารถคลิกใช้งานได้หรือไม่ได้ และด้วยเหตุผลใด เอเจนต์จะต้องรายงานสาเหตุรวมถึงอธิบายแนวทางแก้ไขให้ผู้ใช้ทราบอย่างชัดเจน **นอกจากนี้ ห้ามข้ามการทดสอบหรือเปิดโมดอลแล้วกดปิดเฉย ๆ แต่จะต้องทำจนจบกระบวนการจริง (End-to-End) เช่น การทดสอบและกดสร้างเอกสารบิลรับซื้อทั้งโหมด STOCK และโหมด TRADING ให้สำเร็จจริงในระบบ เพื่อยืนยันว่าการแก้ไขไม่กระทบต่อฟังก์ชันการทำงานหลัก**
* **การจำกัดพื้นที่การทดสอบตามคำสั่ง (Scoped Testing Policy):** การทดสอบระบบจะต้องจำกัดพื้นที่อยู่เฉพาะส่วน/หน้าจอที่ได้รับมอบหมายเท่านั้น (เช่น หากให้ทดสอบบิลรับซื้อ ให้เจาะลึกเฉพาะฟังก์ชันในหน้าจอระบบบิลรับซื้อ) ห้ามออกไปกดหรือทดสอบระบบโมดูลอื่นนอกเหนือจากที่ระบุในสโคปงาน เว้นแต่จะได้รับคำสั่งให้ออดิตทั้งโมดูลหรือทั้งแอปพลิเคชัน
* **การตรวจสอบส่วนประกอบปุ่มและตัวกรองบางส่วน (Tabs, Filters, and Table Click Testing):** บนหน้าจอที่ได้รับมอบหมาย ปุ่มและฟังก์ชันการกรองที่เกี่ยวข้องจะต้องถูกทดสอบการคลิกเพื่อตรวจสอบการทำงาน โดยครอบคลุม:
  - **แท็บส่วนหัวหน้าจอ (Header Tabs):** เช่น แท็บ "บิลรับซื้อ" และแท็บ "ประวัติเปลี่ยนตัว Supplier"
  - **ปุ่มตัวกรอง (Filter Options):** สำหรับตัวกรองประเภทและสถานะ ให้ทดลองกดสุ่มเลือกเพียงบางปุ่มพอเป็นตัวอย่าง (เช่น เลือกประเภท 1 เงื่อนไข และเลือกสถานะ 1 เงื่อนไข) เพื่อยืนยันว่าหน้าจอมีการคัดกรองข้อมูลถูกต้องและเรียกใช้ API ได้จริง ไม่จำเป็นต้องไล่กดสลับทดสอบจนครบทุกปุ่มทุกตัวเพื่อเป็นการประหยัดเวลาและทรัพยากรในการตรวจสอบ
  - **การคลิกเปิดดูรายละเอียดในตาราง (Table Detail & Row Clicks):** ทดลองคลิกแถวในตารางเพื่อดูว่ากล่องเปิดแสดงรายละเอียด (Detail Modal) แสดงข้อมูลขึ้นมาถูกต้อง อ่านง่าย สวยงามเป็นระเบียบหรือไม่
* **การตรวจสอบแบบ Responsive ในโหมดพกพาหลังเดสก์ท็อป (Mobile/Responsive Verification after Desktop):** หลังจากทดสอบในโหมดเดสก์ท็อปเรียบร้อยแล้ว จะต้องจำลองการแสดงผลและทดสอบปุ่ม ตัวกรอง ตารางข้อมูลในขนาดหน้าจอมือถือ (เช่น 375px) เสมอ เพื่อให้มั่นใจว่าปุ่มกดได้ง่าย ตารางถูกแปลงเป็น Card list สวยงามไม่มีการเลื่อนด้านขวา (Horizontal scroll) และข้อมูลทั้งหมดแสดงผลจัดระเบียบเรียบร้อยในสัดส่วนหน้าจอนั้น ๆ
* **การวิเคราะห์จุดบกพร่องและเสนอแนะเพื่อปรับปรุงทางสายตา (Proactive Visual Self-Audit & "Eh" Policy):** ในการรันเบราว์เซอร์และอ่านข้อมูล DOM เอเจนต์ต้องตั้งใจสังเกตและคิดวิเคราะห์ (เอ๊ะ) เพื่อตรวจหาจุดบกพร่องทางสายตา เช่น ข้อความซ้อนทับกัน ตารางที่บีบเกินไปจนอ่านยาก หรือปุ่มที่เว้นระยะไม่สมส่วน หากพบจุดเหล่านี้ ให้จดบันทึก สรุปรายละเอียด และเสนอแนะแนวทางแก้ไข (เช่น การปรับเป็นตาราง auto-fit หรือการขยับ spacing) ให้ผู้ใช้พิจารณาตัดสินใจก่อนลงมือแก้ไขจริงเสมอ เพื่อควบคุมคุณภาพ UI ให้เป็นไปตามสไตล์ความชอบของผู้ใช้และมาตรฐานระบบอย่างสูงสุด
